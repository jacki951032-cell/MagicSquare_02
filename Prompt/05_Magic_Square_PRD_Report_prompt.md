6. **(선택) User Story**  
   - `As a learner I want... so that...` — FR·AC가 이미 있으면 **중복**이 될 수 있어, 교육 과정에서만 요구되면 추가.

## TDD PRD에 **특히** 넣는 것 (요약)

- **입·출력 계약**이 **문자 단위**까지 고정된 부분(오류 `code` / `message`, 성공 `int[6]` 형식).  
- **AC가 테스트로 바로 쓰일** 정도의 구체성(“FR-05 미호출” 등).  
- **Boundary(검증) vs Domain(로직)** 구분과 **서로 먼저/나중** 금지 규칙(이미 8.3).  
- **고정 시나리오/픽스처** 출처(Report/04 등) — 회귀와 Gherkin 혼동 방지.

**Cursor**

Dual-Track **UI + Logic TDD**로 가는 판단은 **합리적**입니다. 핵심은 “경계(사용자나 외부에 **보이는/약속된 것**)”과 “**규칙·계산**”을 나눠 RED를 만들고, GREEN/REFACTOR는 한 스윗에 묶는 것이고, 슬라이드에 나온 **UX Contract / Logic Rule** 어휘는 테스트 이름·Given/Then을 일관되게 쓰기에도 좋습니다.

## 잘 맞는 점

- **RED의 원인이 다름**: UI 쪽은 “화면/접근성/메시지 가시성”에서 실패시키고, 로직 쪽은 “34, 빈칸 2, 시도1·2”에서 실패시키면 **책임이 겹치지 않습니다.**
- **“결정(Decision)이 있는 To-Do만 테스트”**는 좋은 필터입니다. *정리만 하기* 같은 건 단위테스트로 안 가져가는 게 맞고, *거부/반환/가시* 같은 **판단**만 테스트에 태우는 건 TDD 품질을 올립니다.
- **Magic Square 4×4**처럼 규칙이 명확한 도메인이면, 로직 트랙은 `FR-04`/`FR-05` 같은 **불변·계약**과 1:1로 매핑하기 쉽습니다.

## 짚고 가면 좋은 점 (프로젝트와의 정합)

지금 `PRD_Magic_Square_4x4_TDD.md`에는 **실제 UI 앱**은 *Out of Scope*로 잡혀 있고, Boundary는 **콘솔·`pytest`·계약(코드/`message`)** 쪽에 가깝게 서술되어 있습니다. 그래서 “Dual-Track”을 쓰려면 둘 중 하나로 **범위를 맞추는 게** 중요합니다.

1. **진짜 React 등 UI**를 쓰는 경우  
   - PRD의 *In-Scope*에 “**최소 UI**(격자·오류·완성 메시지)**”**를 넣고, **UX Contract**는 그 화면에 대해 (Visible, Disabled, Includes 등) 쓰는 게 자연스럽습니다.  
2. **UI 없이 TDD만** 할 경우  
   - “UI 트랙”을 **Boundary/Presenter**로 번역하는 게 일반적입니다. 즉, **사용자가 아니라 “공개 API·JSON·CLI·에러 DTO**에 대해 `E_*` / `message` / `int[6]`가 *보이는/안 보이는* 수준**으로 Contract 테스트**를 씁니다. (슬라이드의 ①은 “화면”이 아니라 **“외부 계약”**으로 읽기)

슬라이드의 ③ **시나리오 → UX(①) + 로직(②)** 테이블은 그대로 PRD/테스트 추적에 쓰기 좋습니다. 지금 PRD에 있는 **9.1 시나리오**를 그 표 형식으로 한 열씩 늘려 쓰면, 말씀하신 방법론과 1:1로 맞출 수 있습니다.

## 실무적으로의 한 줄 조언

- **로직 트랙**: `MagicSquareJudge`, `DualAssignmentResolver` 같이 **순수 함수/도메인**에만 `Allow/Reject`, `Return`을 걸기.  
- **UI(또는 Boundary) 트랙**: `queryByText`뿐 아니라, **콘솔 과제**면 `result.code`, `result.message`, 화면 문자열이 **“포함/미포함”**인지로 **Visible/Includes**에 대응시키기.

정리하면, **Dual-Track UI + Logic TDD는 방향이 맞고**, Magic Square처럼 규칙이 촘촘한 주제랑 잘 맞습니다. 이 레포를 **웹 UI까지** 쓸지, **PRD대로 콘솔·경계만** 쓸지에 따라 “UI”를 **화면**으로 둘지 **공개 경계(Contract)**로 둘지만 정하면 됩니다. 원하시면 **Out-of-Scope에 UI를 얼마나 넣을지**에 맞춰 PRD 4절·8절 문장만 어떻게 바꾸면 되는지 짧게 초안을 적어 드릴 수 있습니다.

## 2. Executive Summary (재정리)

- **제품**: 4×4, 0이 정확히 2, 시도1→시도2의 **이중 대입** 후 `int[6]` 또는 **고정 `E_*` + `message`**.
- **Dual-Track**: **UI(경계)는 디자인·계약으로 RED**, **로직은 BR·FR로 RED**, **GREEN·REFACTOR는 풀 스윗**으로 항상 동행(8.3, 슬라이드와 동일).
- **MLOps 성격(비-ML)**: *학습* 대신 **“스펙·픽스처·리포·배포(CI) ”의 일관성**—계약·테스트데이터·파이프라인을 **한 세트**로 취급한다.

## 3. UI 트랙 ↔ PRD 요구 (UX Contract)

PRD 4.2는 **웹 UI Out-of-Scope**이므로, **UI = Boundary / 외부 계약**으로 읽습니다(React가 생기면 동일 **어휘**를 화면에 이식).

| PRD | UX Contract로 말하기 (예) |
|-----|----------------------------|
| **FR-01** | “도메인 실패 메시지”는 **not visible** / 경계 `E_*`는 **message가 정확히 includes** (문자 그대로). `FR-05` 미호출은 “해결 시도” 경로 **impossible**로 검증. |
| **FR-05 (경계 맵핑)** | `E_NO_VALID_ASSIGNMENT` / `E_DOMAIN_FAILURE` + 고정 `message` → **visible** + **includes** 1:1. |
| **AC-05-2~3** (성공 6-튜플) | (UI가 있으면) “완성”/숫자 표시가 **visible**; 콘솔/API면 응답 DTO에 **6개 값 포함**. |
| **테스트 ID** | `UI-T-01`~`07` = **Contract-first** (PRD 8.1) |

**Logic과의 분리**: “행 합 34” 같은 **수학**은 **로직**; “`E_INPUT_SHAPE` 때 이 문장”은 **UI/경계**.

## 4. Logic 트랙 ↔ PRD 요구 (Logic Rule)

| PRD | Logic Rule로 말하기 |
|-----|---------------------|
| **FR-02, FR-03** | 좌표·누락 (a,b), a&lt;b: **계산/유지** 규칙, `Calculate` + 불변. |

## 5. MLOps로 PRD를 “운영”에 붙이기 (비-ML 맵핑)

| PRD 절/항목 | MLOps에서 맡는 역할 |
|-------------|---------------------|
| **입·출력 계약** (`code`, `message`, `int[6]`) | **스펙(스키마) / API 버전**: 변경 시 **9.2 변경으로 버전**·**CHANGELOG** (회귀와 동일 메커니즘). |
| **9.3 고정 4×4, Report/04** | **큐레이트된 “평가 데이터” / Golden set**: 팀의 **source of truth**; 재계산 `a,b`·기대 6-튜플. |
| **9.4 Property (Hypothesis)** | **Property-based = 스트레스/탐색 실험**; 시드·반복 횟수로 **재현** 고정(선택). |
| **NFR-01 커버리지** | **CI 품질 게이트** (로직/경계 **다른** 기준: Domain ≥95%, Boundary ≥85%). |
| **NFR-02 결정론** | **“추론/실행” 재현성**과 동일: 동일 `M` → 동일 산출. **의사 난수 없는 도메인**이면 **재현=결정론**으로 끝. |
| **9.1 T-S-01~05, pytest+CI** | **Continuous Validation**: PR마다 **회귀 방지**; 삭제된 테스트 **금지**(9.2). |
| **NFR-04 (50ms)** | **SLO/성능 스모크** (선택): “라인 서비스”가 아니라 **훈련 과제**에서도 지연 **경보**에 해당. |
| **NFR-05 (print 금지)** | **로그/관찰** 표준: 파이프라인에서 **비결정론**적인 콘솔 의존 제거. |
| **10절 Architecture** | **Domain = 순수 (외부 0)**; Boundary = **I/O** — MLOps에서 **모델 서빙 vs 전처리**와 같은 **책임 경계**로 유지. |

**전통 MLOps에서 빠진 것(의도)**: *모델 등록(Registry)*, *재학습(Continuous Training)*, *A/B* — **본 PRD 4.2(생성/일반화/DB 제외)**와 맞다. 나중에 **딥러닝으로 “다음 수 추천”**을 넣으면 그때 **Registry/모니터링(드리프트)** 를 같은 PRD에 “확장”으로 추가하면 됩니다.

## 7. 루프: RED / GREEN / REFACTOR (정리)

1. **UI(경계) RED**: `E_INPUT_SHAPE` 등 “문장·코드”가 스펙과 **불일치**하도록 1건만 실패.  
2. **Logic RED**: `is_magic` / 대입 1가지가 **BR/FR**과 **불일치**하도록 1건만 실패.

## 2. Dual-Track UI + Logic TDD — 정합성

### 잘 맞는 점

| PRD 위치 | 분석 |
|----------|------|
| **1·2절** | “스펙·계약이 통과 기준”이라 **경계(출력 표현) vs 도메인(34·대입)** 이원을 **문제 정의**에서 이미 밀고 감. |
| **FR 분리** | **FR-01 = Boundary**, **FR-02~05 = Domain**이 FR 단위로 **깔끔**함. `FR-05`가 **도메인 실패**와 **경계 `E_DOMAIN_FAILURE`+message**를 같이 쓰는 것도, **이중 매핑(11절)**으로 **리스크를 테이블로 고정**해 둔 상태. |
| **8절** | `UI-T-*` / `D-T-*`, **병렬 RED → GREEN → REFACTOR**, **한쪽만 먼저 금지** — 슬라이드의 **Dual-Track**과 **동일한 운영 규칙**. |
| **10·12절** | `Validator/Presenter` vs `Judge/Resolver` — **테스트 추적**이 **레이어**와 맞음. |
| **9.1 T-S-01~04** | **시도1/2/실패/입력오류**로 **로직·경계** 시나리오가 **한 표**에 있음. |

### 정리한 컨셉과의 **간극(용어·범위)**

- **8.1 제목**이 `Boundary (UI) TDD`인데, **4.2**는 **웹/데스크톱 UI Out-of-Scope**입니다. 실제로는 **“UI = 공개 API·응답 DTO·고정 `message`”**에 가깝습니다. **슬라이드의 UX Contract(Visible/Includes)** 를 **문자·코드 단위**로 쓰려면, PRD에 **“UI = Boundary / 외부 계약; 화면은 선택”** 한 문장이 있으면 **독자 혼동**이 줄어듦.
- **UX Contract 어휘**(Visible, Includes, …)는 **8절·5절**에 **직접 등장하지 않습니다**. `AC`는 **검증 가능**하지만, **팀이 쓰는 동사 셋**으로 **UI-T 테스트 이름**을 맞추려면 **부록 한 표**가 있으면 **Dual-Track 말**과 **1:1**이 됨.
- **FR-04**는 **도메인 핵심**인데, 12절 추적은 **`MagicSquareJudge`**에 잘 붙어 있음. 다만 **FR-04 “0 있으면 사전조건”** (124행)이 **11절**에만 **짧게** 있고, **D-T**에서 **“0 있음 → Reject/Assert”** 중 어떤 쪽을 **AC로 고정**할지는 **구현 선택**에 열려 있어, **로직 트랙 RED**의 **하나의 실패 케이스**를 PRD가 **강제하지는 않음** (의도일 수도, **모호성**일 수도 있음).

### Logic 트랙

- **BR-01~10**이 **FR-02~05**를 **한 줄 규칙**으로 **중복 서술** — **Traceability(12절)** 에 유리하고, **Logic Rule** 언어로 **그대로 옮기기** 좋음.
- **FR-02, FR-03**은 **AC가 “성공 경로” 위주** — **Dual-Track**에서 **로직은 실패/경로가 적다**는 점과 **맞음**; **FR-04·FR-05**에 **부정·경계**가 **집중**됨.

## 3. MLOps — PRD에 이미 “깔린” 것 vs 없는 것

### 이미 PRD에 있는 **운영·품질** 요소 (MLOps에 대응 가능)

| 요소 | PRD |
|------|-----|
| **스펙/스키마 버전** | 9.2 `code`·`message`·6-튜플 변경 = **계약 변경** + **CHANGELOG** |
| **Golden / 평가 데이터** | 9.3 **예 A** 격자, **Report/04** 출처, T-S-02/03 **별도 M**·**검산** |
| **재현성** | NFR-02 **결정론** (동일 `M` → 동일 결과) |
| **CI 품질 게이트** | NFR-01 **커버리지** (Logic vs Boundary **다른 기준**), T-S-05 **pytest·CI** |
| **회귀 정책** | 9.2 테스트 **삭제 금지**, **RED 먼저** |
| **SLO(선택)** | NFR-04 **50ms** |
| **관찰** | NFR-05 **print** 금지, **로깅/캡** |

### PRD에 **없거나** 이름만으로는 **약한** 부분

- **“MLOps”** **용어**와 **CI/CD**가 **NFR-01, T-S-05**에 **흩어져** 있어, **“우리 팀이 말하는 MLOps = 계약+데이터+CI+재현”** **정의**는 **한 블록**에 없음.
- **의존성** (Python, `pytest`, `hypothesis` 여부) **고정**은 **3절**이 **러너만** — **락파일/재현 빌드**는 **MLOps** 맥락에서 **선택**으로 쓸 수 있으나 **PRD에 없음**.
- **“모델/학습/배포”** — **4.2**와 **맞게** **의도적 제외**; 지금 **정리한 MLOps 컨셉(비-ML)** 과 **일치**함.

## 4. 내부 일관성 점검

- **`E_NO_VALID_ASSIGNMENT`(도메인)** vs **`E_DOMAIN_FAILURE`(경계)** — **AC-05-4, 11절, 8.1**에서 **같은 실패**를 **두 층**에서 말함. **Presenter 매핑**이 **UI-T-06**에 **묶여** 있어 **Traceability**와 **맞음**; **신규 독자**는 **“언제 어느 코드가 사용자에게 보이는가”** 를 **5절 한 그림**으로 **보는 게** 이해에 도움 (PRD **보강 포인트**).
- **FR ID**: FR-01만 **(Boundary)** 라벨, FR-02~05는 **Domain이지만** 제목에 **Track 라벨 없음** — **의도적 단순화**로 **읽기** 좋고, **8절**과 **같이 읽어야** **이원**이 **드러남**.
- **12절** `INV-D-*` — **Report/02** **참조**인데, **이 PRD 파일** 안에 **INV 목록**은 **없음**; **같이 보관**하지 않으면 **온보딩**에 **의존**함.

## 6. 결론

- **Dual-Track**: PRD는 **이미** **경계 vs 도메인**·**병렬 TDD**·**추적**까지 **갖췄고**, **“UI”** 를 **슬라이드의 시각 UI**로 **오해하지 않게** **용어( Boundary / Contract )** 만 **정돈**하면 **정리한 컨셉**과 **거의 완전히 정렬**됩니다.  
- **MLOps**: **학습/모델**이 아니라 **계약 버전 + Golden 데이터 + 결정론 + CI/커버리지/회귀** 로 읽을 때, **5·7·9절**이 **역할**을 **이미 나눠 가짐** — 다만 **한 절**에 **이름**과 **다이어그램**이 없어서 **팀内 공유**에는 **“PRD 0절 또는 7절 직후 부록”** 수준 **추가**가 **분석상 이득**이 큼.

원하시면 **문서에 넣을 짧은 문단(0절 Executive + MLOps 5줄 + “UI=Boundary” 3줄)** 초안을 **다음 턴**에 파일에 맞춰 **패치**할 수 있습니다.
