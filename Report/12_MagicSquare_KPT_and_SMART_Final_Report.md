## 1. KPT 요약(한눈에 보기)

### 1.1 Keep (계속 잘한 것)
- **테스트 먼저 만들고 개발한 방식** → 버그를 빨리 찾을 수 있어서 안정적이었음
- **역할 나눔이 잘 된 구조(GUI → Boundary → Domain)** → 어디를 고쳐야 할지 명확해 작업이 수월했음
- **작업 내용을 보고서로 남긴 점** → 나중에 봐도 “왜 이렇게 했는지” 바로 이해 가능

### 1.2 Problem (아쉬운 점)
- **GUI 쪽 자동 테스트가 부족함** → 화면 관련 수정 시 회귀를 늦게 발견할 수 있음
- **한 함수가 너무 많은 일을 함** → 읽기 어렵고 수정 시 다른 부분까지 깨질 위험이 큼
- **비슷한 검증 코드가 여러 곳에 있음** → 규칙 변경 시 여러 파일 수정이 필요해 실수 가능성이 커짐

### 1.3 Try (다음에 해볼 것)
- **GUI 핵심 기능부터 테스트 추가하기** → 리팩토링 전에 안전장치(회귀 방지)를 먼저 확보
- **복잡한 함수 쪼개기** → “검증/실행/결과 만들기”를 분리해 이해·수정 용이성 향상
- **검증 규칙 한곳으로 정리하기** → 중복을 줄이고 규칙 변경이 쉬운 구조로 개선

## 2. KPT 상세

### 2.1 Keep (유지)
- **요구사항/PRD 선정리 후 구현:** 구현 전 기준을 확정해 방향성이 흔들리지 않음
- **테스트 우선 흐름 유지:** RED 스켈레톤을 실제 검증 테스트로 전환하고 **`43 passed`**까지 만든 흐름이 품질 안전망으로 효과적
- **핵심 규칙 테스트 기반 개발:** Magic Square 규칙을 테스트 케이스로 검증하며 개발해 오류를 조기 차단
- **구조/레이어 분리 원칙 유지:** `gui -> boundary -> domain` 단방향 구조, Entity/Control/Boundary 역할 분리, 에러 코드 매핑으로 변경 영향 최소화
- **객관적 품질 확인 유지:** `pytest`와 `coverage`로 구현 결과를 수치와 리포트로 확인
- **산출물 문서화 습관 유지:** 구현/분석 보고서를 남겨 의사결정 근거와 인수인계, 추적성이 강화됨
- **작업 단위 분절(프롬프트 엔지니어링 관점):** **보고서 내보내기 → 브랜치 생성 → 업로드**까지 작업 단위를 잘 끊어 진행

### 2.2 Problem (문제)
- **초반 기준 분리의 어려움:** 요구사항과 테스트 범위를 나누는 데 시간이 소요됨
- **문서/테스트/추적성 비용 증가:** 기능 구현보다 정리 작업에 더 많은 시간이 필요했음
- **GUI 테스트 공백:** `main_window` 이벤트/에러 표시 경로 자동화 테스트 부족으로 UI 리팩토링 시 회귀 위험이 큼
- **책임 집중 함수(SRP 위반 소지):** `boundary.solve`, `matrix_validator`, `domain.solver` 일부가 검증/오케스트레이션/포맷팅을 함께 담당
- **검증 로직 중복 가능성:** `boundary`와 `domain`에 유사 검증이 존재할 수 있어 규칙 변경 시 동기화 누락 위험
- **AI 산출물 검증 루틴 미성숙:** 제안 내용을 그대로 적용하기보다 실행·검토로 확인하는 과정이 아직 익숙하지 않음
- **문서-코드 동시 변경 혼선:** README/PRD/리포트/코드가 함께 변하며 “최종 기준” 확인 비용이 발생
- **요청 전환으로 흐름 단절(프롬프트 엔지니어링 관점):** 중간에 요청이 바뀌며 **(분석 ↔ 구현 ↔ 업로드)** 작업 흐름이 자주 끊김

### 2.3 Try (시도)
- **요구사항·테스트·성공기준을 더 작은 단위로 분해**해 착수 비용과 재작업을 줄인다
- **AI 요청 시 조건/입력/기대결과를 구체화**해 산출물 품질과 검증 효율을 높인다
- **GUI 핵심 시나리오부터 테스트를 우선 보강**한다: `_on_solve_clicked`, `_read_matrix` 중심
- **유스케이스(control/use-case) 계층 도입을 실험**해 오케스트레이션을 분리하고 `boundary`는 I/O 변환에 집중한다
- **검증 파이프라인을 단계별로 정리**하고 공통 규칙/도메인 불변식 경계를 문서+코드로 고정한다
- **작업 단위 체크리스트(DoD)와 To-do 관리**로 진행상태와 품질 확인을 표준화한다
- **문서 단일 기준(SSOT) 지정:** 최종 기준 문서 1곳을 정하고 나머지는 링크/요약으로 유지한다
- **세션 초반 “3줄 고정 템플릿”으로 시작(프롬프트 엔지니어링 관점):** 오늘 목표/금지사항/최종 산출물을 3줄로 고정해 요청 전환에도 흔들리지 않게 한다

## 3. SMART 목표(다음 사이클 실행 계획)

### Goal 1: GUI 회귀를 막는 핵심 시나리오 테스트 확보
- **S (Specific):** `main_window`의 입력→solve→성공/에러 표시 흐름을 자동화 테스트로 커버
- **M (Measurable):** 핵심 시나리오 테스트 **최소 3개** 추가, 전체 테스트 **`43 passed` 유지**
- **A (Achievable):** 기존 `pytest` 기반으로 시나리오만 확장
- **R (Relevant):** UI 리팩토링 시 회귀 리스크를 최우선으로 감소
- **T (Time-bound):** **이번 주 내** 완료

### Goal 2: `solve` 책임 분리(Use-case 계층 도입)로 SRP 개선
- **S:** 오케스트레이션을 `use_case/control`로 이동, `boundary.solve`는 I/O 변환 중심으로 재정렬
- **M:** 테스트 전부 통과 유지, `boundary.solve`의 역할이 명확해지고 복잡도(분기/길이)가 감소
- **A:** 레이어 원칙(`gui -> boundary -> domain`)을 유지하며 점진 적용
- **R:** 변경 영향 축소 및 유지보수성 향상
- **T:** **2–3일 내 1차 완료**

### Goal 3: 검증 로직 단일화 + 작업 완료 기준(DoD) 루틴 고정
- **S:** 공통 규칙/불변식 경계를 한 곳으로 수렴, 검증 파이프라인을 단계별 함수로 분리
- **M:** 중복 검증 지점 **최소 1곳 제거**, 작업 3건 연속 **DoD 누락 0건**
- **A:** 기존 `matrix_validator`/domain 검증을 재배치·추출하는 방식으로 진행
- **R:** 규칙 변경 시 동기화 누락 방지, 품질 설명력 강화
- **T:** **1주 내** 완료

### Goal 4: 세션 초반 “목표/금지/산출물 3줄” 템플릿으로 흐름 단절 최소화
- **S:** 모든 세션 시작 시 아래 3줄을 고정하고, 중간에 요청이 바뀌어도 이 3줄을 기준으로 재정렬
- **M:** 세션 중 요청 전환(분석↔구현↔업로드) 발생 시에도 최종 산출물 누락 0건(체크리스트로 확인)
- **A:** 템플릿 복붙으로 즉시 적용 가능
- **R:** 컨텍스트 스위칭 비용과 재확인 시간을 줄여 생산성 향상
- **T:** **다음 세션부터 즉시 적용(1주간 유지)**

**3줄 고정 템플릿**
- **오늘 목표:** (예: GUI 테스트 3개 추가 + 리포트 1개 갱신)
- **금지사항:** (예: 큰 리팩토링 금지 / 범위 확장 금지 / 미검증 코드 커밋 금지)
- **최종 산출물:** (예: PR 1개 + 테스트 통과 로그 + 보고서 링크)

## 4. 개인 학습 성과
- **TDD 경험:** RED → GREEN → REFACTOR 흐름을 실제로 수행
- **규칙 기반 구현 역량:** Magic Square 규칙을 분석해 코드로 구현
- **테스트 도구 숙련:** `pytest`로 단위 테스트 작성/실행, `coverage`로 객관적 확인
- **품질 설명력 강화:** 요구사항 ↔ 테스트 ↔ 결과(리포트)가 연결돼야 품질을 설명할 수 있음을 학습
- **AI 활용 역량:** “생성”보다 **실행·검증·추적**이 핵심이라는 점을 체득

## 5. 팀 차원의 개선 액션 아이템
- **착수 전 요구사항/성공 기준 1페이지 정리**(간단 명료하게)
- **핵심 테스트 선작성** 후 구현 시작
- **AI 제안물은 실행·리뷰 후 채택**(검증을 프로세스로 고정)
- **제출물(README/PRD/테스트 결과/리포트) 크로스체크** 후 제출
- **작업을 작은 단위로 분해 + To-do로 진행 관리**
- **담당자/마감일 지정**으로 책임 범위 명확화
