## Keep (유지할 점)

- **작업 방식/프로세스:** 작업 단위를 잘 쪼개서 **(보고서 내보내기 → 브랜치 생성 → 업로드)** 진행했고, **Ask → Plan → Agent** 흐름으로 Cursor를 체계적으로 활용했음. PRD·요구사항을 먼저 정리한 뒤 구현한 점도 좋았음.
- **테스트 중심 개발:** 구현 전에 TDD로 **RED 테스트를 먼저 만들고**, 핵심 규칙을 테스트 케이스로 검증하면서 개발해 버그를 빨리 발견하고 안정성을 높였음. `pytest`·`coverage`로 결과를 객관적으로 확인했음.
- **구조/설계:** **GUI → Boundary → Domain**(또는 Entity·Control·Boundary)로 역할을 명확히 분리해 수정 지점이 분명하고 이해가 쉬운 구조를 만들었음.
- **문서화/추적성:** 작업 내용을 보고서로 남기고, 기능 구현뿐 아니라 문서화와 테스트 추적성까지 함께 정리해 나중에 의사결정 이유를 빠르게 파악할 수 있었음.
- **학습/적용:** 다양한 프롬프트 엔지니어링 전략을 배우고 실제로 적용해본 경험이 의미 있었음.

## Problem (아쉬운 점)

- **요구사항/테스트 경계 설정이 어려움:** 초반에 요구사항과 테스트 범위를 정확히 분리·정의하는 데 시간이 많이 들었고, 특히 RED(실패 조건)를 코딩 전에 명확히 못 정해 흐름이 흔들렸음.
- **문서·테스트·추적성 비용이 큼:** 기능 구현보다 문서화, 테스트 작성, 변경 추적(Traceability) 정리에 더 많은 시간이 들었음. 작업 중 문서와 코드도 함께 바뀌어 최종 기준(싱글 소스 오브 트루스) 확인에 추가 시간이 발생.
- **AI 활용의 검증 단계 미숙:** AI 제안을 그대로 쓰기보다 검증이 필수인데, 테스트 검증 없이 AI 결과에 과도 의존하는 문제가 있었고 아직 이 검증 루틴이 익숙하지 않았음.
- **작업 흐름 단절/변경 추적 어려움:** 중간에 요청이 분석↔구현↔업로드로 바뀌며 컨텍스트 전환이 잦았고, GREEN 단계에서 한 번에 많은 코드 생성으로 변경사항 추적이 어려웠음.
- **테스트 커버리지 및 GUI 자동 테스트 부족:** 에이전트 기반 작성 시 커버리지가 기대보다 낮았고, 특히 GUI 자동 테스트가 부족해서 화면 수정 시 회귀(기능 깨짐)를 늦게 발견할 위험이 큼.
- **구조적 품질 이슈(리팩터링 포인트)**:
  - 하나의 함수가 너무 많은 역할 → 가독성 저하, 수정 시 사이드이펙트 위험 증가
  - 유사 검증 로직의 중복 → 규칙 변경 시 다중 파일 수정 필요, 실수 가능성 증가

## Try (다음에 해볼 것)

- **리팩토링 전 안전장치부터:** GUI 핵심 기능 중심으로 테스트를 먼저 추가하고, 테스트를 더 자주 실행해 변경 영향(coverage 포함)을 빠르게 확인한다.
- **복잡도 낮추기:** 큰 함수를 **“검증 / 실행 / 결과 생성”**으로 분리하고, 검증 규칙은 한 곳으로 모아 중복을 줄이며 규칙 변경을 쉽게 만든다.
- **변경 이해도 강화:** 다음 단계로 넘어가기 전에 변경 내용을 더 충분히 이해하고, AI 생성 코드의 변경 사항을 잘 추적할 수 있도록 추가 체크를 도입한다.
- **세션/작업 프로세스 표준화:** 세션 초반에 오늘 목표·금지사항·최종 산출물을 3줄 템플릿으로 고정하고, 구현 전 요구사항/테스트케이스/성공기준을 더 잘게 쪼개 정리한다.
- **AI 요청 품질 개선:** AI에게 요청할 때 조건·입력값·기대 결과를 더 구체적으로 작성한다.
- **마무리 습관:** 기능 구현 후 테스트 결과·coverage·남은 위험을 짧게 요약한다.
- **다음 프로젝트 운영:** 작업 단위별 간단한 체크리스트로 진행 상황을 상시 확인한다.
