# 4x4 Magic Square 문제 정의 보고서

## 개요
본 문서는 4x4 마방진 프로그램 개발에 앞서, 설계와 구현 이전에 수행한 문제 인식 과정을 정리한다.  
핵심 목적은 "정답 산출"이 아닌, 제약 기반 문제를 시스템적으로 정의하고 검증 가능한 형태로 구조화하는 데 있다.

## STEP 1 - Observation (관찰)

### 1) 현재 우리가 해결하려는 상황은 무엇인가?
현재 상황은 4x4 격자의 16개 칸에 값을 배치할 때, 여러 제약이 동시에 만족되어야 하는 상태를 다루는 것이다.  
이 문제는 단순 채우기 작업이 아니라, 각 선택이 이후 가능 상태를 바꾸는 제약 결합형 탐색 상황이다.

### 2) 왜 4x4 마방진 문제를 다루는가?
- 3x3 대비 제약 상호작용이 커져 체계적 정의가 필요하다.
- 너무 큰 규모는 아니어서 모델링과 검증 연습에 적합하다.
- 알고리즘적 사고(탐색, 가지치기, 검증)를 실습하기 좋은 중간 난이도다.

### 3) 어떤 학습/시스템 설계 맥락에서 등장하는가?
- 학습 맥락: CSP, 백트래킹, 상태 표현, 제약 검증 훈련
- 시스템 설계 맥락: 입력/검증/탐색/출력 책임 분리
- 엔지니어링 맥락: 문제 정의 -> 제약 명세 -> 성공 기준 -> 테스트 가능성

## STEP 2 - Why #1

### 질문
왜 마방진을 완성해야 하는가?

### 가정 A
"정답이 있는 퍼즐을 끝까지 해결해 성취를 얻기 위해서."

### 가정 A에서 드러나는 불편함/구조적 문제
- 정답 중심 편향으로 과정의 가치가 약화된다.
- 한 번의 해답에 머물러 재사용 가능한 구조가 남지 않는다.
- "완성" 기준이 모호해 품질 정의가 불안정해진다.
- 맞았는지 여부만 남고, 왜 맞는지의 검증 체계가 약해진다.

## STEP 3 - Why #2

### 질문
왜 단순 계산이 아니라 프로그램으로 구현하는가?

### 관점별 답변
- 반복 가능성: 같은 조건을 일관된 절차로 재실행할 수 있다.
- 검증 자동화: 규칙 통과 여부를 즉시, 객관적으로 판정할 수 있다.
- 오류 방지: 수작업 실수(중복, 누락, 계산 착오)를 줄일 수 있다.
- 규칙 기반 사고 훈련: 조건 분해와 명시적 검증 사고를 강화한다.

## STEP 4 - Why #3 (핵심 발견)

### 질문
왜 이 문제를 TDD 방식으로 설계하려 하는가?

### 핵심 발견
TDD는 정답을 빨리 찾기 위한 방식이 아니라, 제약이 많은 탐색 문제를 안정적으로 통제하기 위한 방식이다.

### 무엇이 통제되어야 하는가?
- 상태 전이의 일관성
- 유효성 판정 로직
- 실패/되돌리기 기준
- 종료 조건

### 어떤 불변 조건이 존재하는가?
- 값의 범위/중복 규칙
- 상태 구조의 일관성(4x4)
- 완성 상태의 합 제약 만족
- 부분 상태에서의 정합성 유지

### 왜 입력/출력이 명확해야 하는가?
테스트는 명확한 계약을 필요로 한다.  
입력/출력이 모호하면 통과/실패 기준이 흔들리고, 리팩터링 중 의미 보존이 불가능해진다.

## STEP 5 - 진짜 문제 정의

### 1) 표면 문제 정의 (잘못된 정의)
"4x4 마방진 정답 하나를 만들어내는 것."

### 2) 개선된 문제 정의 (정확한 정의)
"4x4 상태를 제약 기반으로 표현하고, 상태 전이마다 불변 조건을 검증하며, 명확한 입력/출력 계약 하에서 유효한 해를 재현 가능하게 도출하는 프로그램을 설계/검증하는 것."

### 3) 이 문제의 핵심 Invariant
- 도메인 불변성: 허용 값 범위와 중복 규칙 유지
- 구조 불변성: 4x4 상태 표현 일관성 유지
- 제약 불변성: 완성 시 행/열/양 대각선 합 동일
- 부분 정합성: 중간 상태에서도 위반 상태 배제
- 판정 결정성: 동일 입력에 동일 검증 결과

### 4) 실제로 훈련하려는 사고 능력
- 문제 구조화 능력(요구 -> 제약/상태/전이)
- 불변 조건 중심 사고
- 검증 가능성 설계(I/O 계약 기반)
- 탐색 통제 능력(규칙 기반 시행착오 관리)
- 일반화 능력(타 제약 문제로 확장)

## 결론
본 과제의 본질은 퍼즐 완성 자체가 아니라,  
복합 제약 문제를 명확한 명세와 불변 조건 위에서 설계/검증하는 엔지니어링 사고를 훈련하는 데 있다.

## STEP 6 - 요구사항 명세 초안 (입력/출력/실패 조건)

### 1) 입력(Input) 명세 초안
- 격자 크기: 고정값 4x4
- 값 집합: 1~16 정수
- 입력 형태(초안): 4x4 2차원 배열 또는 16개 길이의 1차원 배열
- 미입력 칸 표현(선택): `null` 또는 `0` (한 가지 방식으로 고정 필요)
- 전제 조건:
  - 숫자는 정수여야 한다.
  - 허용 범위를 벗어나는 값은 입력 불가다.
  - 동일 숫자의 중복 입력은 허용하지 않는다.

### 2) 출력(Output) 명세 초안
- 성공 시:
  - 완성된 4x4 격자
  - 검증 결과(유효성: true)
  - 선택적으로 탐색 통계(시도 횟수, 백트래킹 횟수)
- 실패 시:
  - 실패 코드(입력 오류 / 해 없음 / 내부 예외)
  - 실패 사유 메시지
  - 검증 결과(유효성: false)

### 3) 실패 조건(Failure Conditions) 명세 초안
- 입력 형식 오류:
  - 길이가 16이 아님
  - 4x4 구조 불일치
  - 비정수 값 포함
- 도메인 오류:
  - 값 범위 벗어남(1~16 외 값)
  - 중복 값 존재
- 제약 위반:
  - 완성 상태에서 행/열/대각선 합 불일치
  - 부분 상태에서 명백한 불가능 상태 발생
- 탐색 실패:
  - 모든 가능 상태 탐색 후 유효 해를 찾지 못함

### 4) TDD 관점의 최소 테스트 단위 초안
- 입력 검증 테스트:
  - 정상 입력 통과
  - 형식/범위/중복 오류 검출
- 제약 검증 테스트:
  - 완성된 올바른 격자 통과
  - 행/열/대각선 위반 격자 실패
- 탐색 동작 테스트:
  - 해가 있는 경우 유효한 4x4 출력
  - 해가 없는 경우 실패 코드 반환
- 결정성 테스트:
  - 동일 입력에서 동일 판정/동일 결과 보장

### 5) 명세 잠정 결론
입력/출력/실패 조건을 먼저 고정하면, 구현 방식(백트래킹, 휴리스틱 등)이 바뀌어도
시스템 계약은 유지된다. 이는 TDD에서 리팩터링 안정성을 확보하는 핵심 기반이다.
